# Multi-select project filter with name-based environment intersection

The global filter (§9) scoped telemetry to a single Project via an "All projects" sentinel, and the environment bar was per-Project — blank whenever "All projects" was active, so there was no way to filter by environment across the installation (#76). We replace the single-Project selector with a **multi-select**: the chips are additive, an empty selection means all Projects (preserving the existing wire contract — the `project` param is simply omitted), and a non-empty selection sends repeated `project=` params that the four list endpoints (`IssueController.issues`, `LogController.logs`/`tail`, `TraceController.traces`) read as `@RequestParam List<Long>` and apply as `project_id IN (...)`. The environment bar then shows the **intersection of [[Environment Name]]s across the in-scope Projects** — the selected set, or all Projects when the selection is empty — computed server-side by a new `GET /projects/environments?project=…` endpoint (`GROUP BY name HAVING count(DISTINCT project_id) = …`). When the Project selection changes, an active environment filter is **pruned** to the names still in the new intersection rather than cleared, preserving intent whenever it stays valid.

## Considered options

- **Union vs. intersection of environments.** Union (every distinct name in any in-scope Project) reads more useful on paper, but the intent of the bar is "show `production` across the Projects I care about together"; intersection is what "overlapping" means and what a multi-select makes controllable. Chosen: intersection.
- **Strict intersection vs. ignore env-less Projects.** A Project that has never ingested telemetry has no `environment` rows. Counting it strictly makes the intersection empty the moment any such Project is in scope — the "I created an empty Project and all my environment chips vanished" surprise, which would fire constantly on the default all-Projects view. We treat an env-less Project as *no evidence*, not *evidence of absence*, and exclude it from the denominator (`count(DISTINCT project_id) FROM environment WHERE project_id IN (:ids)`).
- **Multi-select vs. keeping the single "All projects" sentinel.** The minimal read of #76 is "show the all-Projects intersection under the existing sentinel." We chose the larger multi-select because the sentinel's strict-intersection default is almost always empty, whereas a selection lets the user scope the intersection to a meaningful set. This is the more expensive path (URL contract, backend endpoints, model types) and is accepted deliberately.

## Consequences

- **URL contract changes**: `?project=5` becomes repeated `?project=5&project=7`; `GlobalFilters.project: number | undefined` becomes a `number[]`, and the three `*Filters` model interfaces and `query-params.ts` builders follow.
- **The default all-Projects view usually shows an empty environment bar**, because few installations have an Environment Name present in *every* Project. This is the accepted cost of a single, uniform rule (bar = intersection of in-scope Projects) with no special-casing of the empty selection.
- **Releases cannot consume a plural Project** — a Release and its Artifacts belong to exactly one Project+version, and `ReleaseController` takes a required single `project`. Releases therefore stops reading the global filter and owns a **page-local single-Project selector**. Hiding the now-inapplicable global bar on Releases (and other such pages) is left to #78; this ADR only moves Releases off the shared filter so the multi-select can land without breaking it.
