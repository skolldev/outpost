# UI feedback goes through the Feedback seam

User-facing success and error messages are emitted through a single injectable seam, `core/Feedback`, which wraps the vendored Spartan sonner `toast` API; components must not import from `@spartan-ng/brain/sonner` directly. This deliberately deviates from Spartan's documented usage — which calls `toast()` at each call site — because a seam gives one home for policy (error toasts persist for 10s, success toasts use the default lifetime) and one point to assert in tests (specs provide `Feedback` as a spy, rather than spying an imported module binding or rendering sonner's CDK overlay under Wallaby). The trade-off is one layer of indirection over a library that is already an abstraction; we accept it because the alternative scatters the sonner dependency and the duration policy across every page and makes the Wallaby-only test suite brittle. The seam is intentionally minimal (`success(message)` / `error(message)`); a richer surface such as a `description` slot is added only when a real second case needs it. Toasts are reserved for transient action outcomes: persistent affordances (the one-time API Token reveal, copy-button state), detailed per-row results (uptime and channel test results), and inline (client-side) form validation stay in their templates. Server outcomes of a form submission — including the login auth error — go through this seam, not inline (see ADR 0008).
